    Defuddle extracts the main content from web pages. It cleans up web pages by removing clutter like comments, sidebars, headers, footers, and other non-essential elements, leaving only the primary content.
    
    I like efficiency and not gobbling up text I don't need sounded like a good idea to me, so I hooked it into Hermes as a (hopefully) install-it-and-forget-it plugin, since I'm also wanting to familiarize myself with how to customize and extend my beloved agent. 🫶🏻
    
    https://github.com/annalinneajohansson82/hermes-defuddle
    
    It shells out to the locally-installed defuddle CLI to fetch article content from URLs. Clean markdown, rich metadata, no API key.
    
    Features:
    
    Clean markdown — strips nav, sidebars, ads, and other boilerplate
    
    Rich metadata — title, description, domain, language, word count, author, publish date
    
    No truncation — returns full page content. 
    Note: the Hermes dispatcher's LLM-processing stage may still summarize or compress long content when an auxiliary model is available. 
    
    Parallel extraction — multiple URLs fetched concurrently via asyncio.

    Heads up — this is a really nice idea, but as-is it'll hit a dispatcher bug that keeps it from actually being used. If your plugin registers its provider under a custom name (e.g. `defuddle`) and someone sets:
    ```
    web:
      extract_backend: defuddle
    ```
    Hermes Agent will **silently ignore it** and fall back to whatever other backend gets auto-detected (Tavily, Exa, etc. — whichever API key happens to be set in the environment). No error, no warning — it just quietly uses a different provider than the one configured.
    
    I ran into the exact same issue trying to wire up a local extraction plugin (`pageprune`) and filed it here: https://github.com/NousResearch/hermes-agent/issues/57581
    
    Root cause in short: `web_extract()` in `tools/web_tools.py` still resolves `web.extract_backend` through a legacy helper (`_get_extract_backend()` → `_is_backend_available()`) that only recognizes a hardcoded list of built-in backends (`exa`, `parallel`, `firecrawl`, `tavily`, `searxng`, `brave-free`, `ddgs`, `xai`). Any plugin-registered provider name outside that list fails the check and gets dropped, even though it's correctly registered in the plugin registry. The registry itself (`agent/web_search_registry.py`, `get_active_extract_provider()`) already handles plugin providers correctly — the dispatcher just doesn't call it first.
    
    So `hermes-defuddle` will register fine and show up in the logs as "registered web provider," but it won't actually get dispatched to unless you either patch that allow-list locally or wait for the fix from the issue above. Worth flagging in your README so people don't think it's silently broken on their end. 
    
    Happy to help test once a fix lands.
