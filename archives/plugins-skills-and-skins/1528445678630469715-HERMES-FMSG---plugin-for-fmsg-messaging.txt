    I've just published **hermes-fmsg**, a plugin which connects Hermes Agent to use fmsg, allowing **agent-to-human, agent-to-agent, and human-to-agent messaging at scale**.
    
    👉 **Try it out:** https://github.com/markmnl/hermes-fmsg
    
    fmsg is a new messaging protocol built on structured threads. Addresses look like `@alice@example.com`, and service providers can support sub-accounts that are perfect for agents, for example: `@alice_hermes@example.com` or you could self-host and have however many addresses and message quotas your server can handle - like email fmsg is domain distributed you can message other hosts at other domains.
    
    > Set `FMSG_HOME_CHANNEL=@your@address.com` for comms from hermes-gateway
    
    fmsg is something I've been working on for years, and I'm publishing it here first because I'm using and loving Hermes!
    
    * **Self-hostable**: self-host an fmsg host on your own domain and choose your own storage, quotas etc. Your messages are handled by you, ensuring data sovereignty and privacy. There is OSS stack: fmsg-docker, but no open UI yet (through there is a CLI); otherwise im running the first fmsg provider over at https://fmsg.io where its all free right now just with message quotas and up to 5 sub-accounts..
    * **A durable agent identity**: people and other agents can contact a stable `@agent@domain` address without sharing a human login.
    * **Context-native threads**: fmsg's message tree maps to Hermes sessions and branch context instead of flattening every conversation into one chat.
    * **Federated reach**: addresses on independently operated fmsg hosts can communicate without going via a 3rd party.
    * **Rich workflows**: text, attachments, reply-all, cron delivery, and agent-to-agent messaging all use the same address.
    
    This is the first time I'm publishing the fmsg white paper discussing the motivations of fmsg - any questions please ask!
