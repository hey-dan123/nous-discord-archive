    Hey Gang
    
        I've put together a little open‑source tool called GitRadar that I've been using to keep my Hermes Agent setup sharp.
    
        What it does:
        - Scans GitHub for new/trending repos (by topic, language, or recency)
        - Filters out noise (tutorial repos, awesome‑lists, dead projects, etc.)
        - Scores each repo on relevance (stars, recency, language/framework fit, licence, topic bonus)
        - Labels them as ADOPT, EXTRACT, FORK/PRODUCT, PLUGIN/SKILL, INSPIRATION, or NOISE
        - Self‑tunes its star‑threshold over time based on the signal‑to‑noise ratio it sees
    
        I’ve wired it into my own Hermes workflow so it continuously feeds me high‑signal repos—I pull the best ideas, patterns, or tools from its output and fold them into my setup, making it more efficient and tuned to what I actually need.
    
        If you’d like to give it a try (it’s just Python, zero deps), the repo is here:
        https://github.com/Sahil-SS9/gitradar
    
        Feel free to fork, tweak the stack preferences to match your own tools, or just run the two scripts and see what pops up. Happy to hear any feedback or ideas for improvement!

    New changes = 
    What
    Adds a configurable spam_name_patterns noise rule, matched against the repo name, to catch star-farmed spam that the existing description-keyword filter misses.
    
    Why
    Farmed repos with date/version-suffixed names (e.g. FL-Product-Version-26, WorpGPT-Latest-2026-AllPrompts, DeepFake-AI-2026-RealTime) and cracked-software bait often carry a clean-looking description, so they slip past noise_description_keywords and waste a slot in the daily output.
    
    How
    New spam_name_patterns default in DEFAULT_THRESHOLDS and the spam_name rule in build_noise_patterns / NOISE_ORDER.
    Patterns are high-precision: they match year/version suffixes and crack/keygen/nulled bait, but are deliberately conservative so legitimate names like llama-3, gpt-4, next.js are not caught.
    Configurable via thresholds (schema updated with the new field).
    New test covers both positive (spam caught) and negative (legit names untouched) cases.

    🚀 GitHub Radar v4.1 is live
    
        Two modes, better signal, less noise:
    
        Daily mode (default) — new-find scan with cache dedup + trending as primary source. No more silent "0 repos" — output now shows cache stats so you know the system is working.
    
        Weekly re-evaluation (--mode weekly) — full cache re-scan that compares pushed_at timestamps to flag repos with NEW activity since last check. Found 59 repos with fresh pushes in the first test run.

        • 4 new query templates (React Native, Flutter, voice, sports)
        • Proper GitHub Search API rate limiting (30 req/min, no more silent failures)
        • pushed_at tracking in cache — repos with new pushes get re-scored
        • Self-tuning lowered star threshold to 50 (signal was 93% — we were filtering too aggressively)
