    https://github.com/yantrikos/yantrikdb-hermes-plugin
    
    I have created the plugin some times back. Now thought to share here since <@387972437901312000> suggested to post here.
    
    yantrikdb-hermes-plugin is based on YantrikDB which is a cognitive persistent memory for AI.
    
    **For memory it supports:**
    
    **remember **- store with importance + domain metadata
    **recall **- semantic search ranked by relevance × recency × importance
    **forget **- tombstone (soft-delete) by memory ID
    **think **- bounded maintenance pass: canonicalise, conflict-scan
    **conflicts **- list open contradictions for the agent to resolve
    **resolve_conflict **- close via keep_winner / merge / keep_both / dismiss
    **relate **- record knowledge-graph edge between two entities
    **stats** - operational snapshot per namespace
    
    **Skills / procedural memory (opt-in, v0.3.0+)**
    skill_define / skill_search / skill_outcome for agent-authored procedural patterns
    Disabled by default - tool schema unchanged unless YANTRIKDB_SKILLS_ENABLED=true
    Stored in shared skill_substrate namespace, server-side schema validation reproduced client
    
    **Install + integration**
    
    pip install yantrikdb-hermes-plugin then yantrikdb-hermes install <hermes_root> — one-shot setup
    Namespace = {base}:{agent_workspace}:{agent_identity} for per-identity isolation
    hermes memory status reports plugin available + ready

    Hi <@329427821208535040> , shipped both of your asks in v0.4.5 (just hit pypi). thanks for catching them, both were real gaps.
    
    one-command install via hermes plugins install now works:
    
    
    hermes plugins install yantrikos/yantrikdb-hermes-plugin
    pip install yantrikdb
    hermes config set memory.provider yantrikdb
    
    hermes drops the plugin source at ~/.hermes/plugins/yantrikdb/ automatically. pip install yantrikdb is the separate step because hermes' plugins-install doesn't auto-pull pip deps yet — that's the engine (~10MB).
    
    on the venv question — yes, pip-install yantrikdb into whatever python env hermes runs from. if you're on pipx install hermes-agent, it's:
    
    
    pipx inject hermes-agent yantrikdb
    regular venv → source it first, then pip. if hermes memory status shows not available ✗ after install, the usual cause is the engine pkg landed in a different python than hermes uses — which hermes && which python confirms. README has both options written up now: https://github.com/yantrikos/yantrikdb-hermes-plugin#install-default--embedded-backend
    
    the existing pip install yantrikdb-hermes-plugin + yantrikdb-hermes install path still works if you'd rather have everything land via pip in one shot — same plugin, different drop location.

    Hello, does yantrikdb-server support embedded models other than the built-in ones? I noticed that the yantrikdb.tom configuration file includes bundled and builtin options.

[2026-05-20T12:21:03.621000+00:00] spranab (id=1506632851242618930)
    The [embedding] section accepts three values for strategy:
    
    builtin (default) — fastembed + ONNX Runtime + Qdrant/all-MiniLM-L6-v2-onnx (384-dim). Downloads the model from HuggingFace on first start, needs an onnxruntime shared library at runtime.
    bundled — the engine's potion-base-2M static embedder (64-dim). Zero network, zero ONNX dep, ~7 MB baked into the binary via include_bytes!. Quality is ~89% of MiniLM at recall@5. This is the default for the Docker image since v0.8.16. Best for air-gapped / edge deployments.
    client_only — no server-side embedder; clients compute embeddings on their end and pass them on every /v1/remember and /v1/recall via the embedding field.
    
    [embedding]
    strategy = "builtin"    # or "bundled" or "client_only"
    dim = 384               # must match whichever embedder you pick
    Other models? Not via config today. builtin is hardcoded to MiniLM-L6-v2; bundled is hardcoded to potion-base-2M. If you want BGE / jina-embeddings / gte-* / any other model, use strategy = "client_only" and embed on your side — the HTTP API fully supports embedding: [f32; N] on /v1/remember and /v1/recall. Whatever dim you set must match what you POST.
    
    Source: crates/yantrikdb-server/src/config/mod.rs#L172-L192.
    
    Pluggable embedders via config (e.g. point strategy = "onnx" at a custom model_path = ...) would be a reasonable feature request if there's demand — would need someone to file an issue at https://github.com/yantrikos/yantrikdb-server/issues.

[2026-05-29T05:43:57.854000+00:00] spranab (id=1509794409523449886)
    v0.4.17 of yantrikdb-hermes-plugin is out.
    
    Before this release, when an agent crystallized a skill via yantrikdb_skill_define, the body was correctly stored but the fact of learning was silent — no future session knew the skill existed unless it happened to call skill_search with the right query.
    
    v0.4.17 closes the loop. On a successful skill_define, the plugin persists (skill_id, type, ts) to $HERMES_HOME/yantrikdb-recent-skills.json. The next session's system_prompt_block() surfaces a "Recently learned skills" block, so the incoming model sees its own prior learning the moment it boots.
    
    Also new: yantrikdb_recall results now expose the per-component score breakdown (similarity / recency / importance / decay → contributions → total). The engine always returned it; the plugin was dropping it on the way out. No more opaque ranking.
