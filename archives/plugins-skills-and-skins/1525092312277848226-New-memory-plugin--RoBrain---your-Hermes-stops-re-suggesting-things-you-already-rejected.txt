    Most memory gives your agent recall. RoBrain gives it vetoes: it captures what you decided and what you ruled out and why.
    
    Real exchange from my own project this morning (screenshot below). Months of sessions, nobody ever hand-wrote this down. It didn't just refuse — it cited the recorded reason and redirected the cost question to the actual constraint. 
    
    What the plugin does
    
    📥 Passive capture — turns ship to your RoBrain server, which extracts decisions (and vetoes) server-side; the plugin needs no LLM key of its own
    📤 Recall — project decision summary in the system prompt + per-turn retrieval, rejections rendered as explicit REJECTED: x — reason warnings
    🔧 robrain_search tool — the agent checks decision history before committing to an approach
    🛟 Fail-open — RoBrain down means degraded recall, never a blocked agent
    Fully self-hosted — your own Postgres, Apache-2.0, multi-arch image, optional fully-local LLM mode (Ollama/LM Studio/vLLM). The same store captures from Claude Code, Cursor, Copilot, and Codex CLI, so your Hermes shares one decision memory with every tool you code in.
    
    Install — three commands, no clone (ships as a standalone plugin per hermes-agent's contribution policy):
    
    npx robrain@latest up               # Postgres + Perception from ghcr.io
    npx robrain@latest install --hermes # plugin → ~/.hermes/plugins/robrain
    hermes memory setup                 # select "robrain"
    Receipts over claims: the repo ships VetoBench, a reproducible benchmark of exactly this failure mode — without decision memory, the agent re-proposed a rejected approach in up to 9/10 tasks; with vetoes in context, 0/50 across five archived runs, every retrieved context committed in-repo. 23 plugin tests + a tarball-install CI gate, verified end-to-end against a live deployment.
