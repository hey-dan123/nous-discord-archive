    Hey everyone  I’ve been working on Maia, a fork from Hermes Agent for teams that want to use capable agents through Slack, Discord, Mattermost, and other company chat systems without giving every user the same authority.
    
    The main experiment is moving security decisions outside the model:
    
    • Identity comes from the authenticated gateway sender (platform:user_id), not from claims in a prompt.
    • File access is deny-by-default and checked again on every operation, with grants by role, team, or user.
    • Conditional edits and governed commands are handed to an authorized reviewer in the shared conversation.
    • Terminal/code execution only runs with authorized paths mounted into an isolated runtime. If that boundary is unavailable, Maia fails closed.
    • Allows, denials, reviews, and executions go to an append-only audit log, with optional SIEM export.
    • Models are replaceable: cloud APIs, OpenAI-compatible endpoints, or local inference.
    
    Repo: https://github.com/and270/maia
    
    It’s still early. Maia does not provide SSO, a VPN, or a zero-trust proxy; it integrates with an existing identity-aware access layer. I’m also not claiming that these controls alone make a deployment compliant.
    
    I’d especially value feedback from people already running Hermes for teams: are these identity, approval, and failure boundaries understandable? Which pieces would be useful upstream in Hermes, and which should remain distribution-specific?
