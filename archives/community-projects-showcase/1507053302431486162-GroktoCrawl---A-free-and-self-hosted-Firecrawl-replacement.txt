[2026-06-06T15:16:57.388000+00:00] magnus919 (id=1512837710665224303)
    The GitHub adapters for GroktoCrawl now handle every common github.com URL pattern — raw files, blob viewer pages, repo roots with READMEs, directory trees, issues, pull requests, discussions, releases, and commits — through a three-tier fallback chain: GitHub's GraphQL API first (single round trip per resource, even with hundreds of comments/reviews), then REST API, then HTML page scrape as a last resort. File URLs target raw.githubusercontent.com directly for the cheapest possible fetch. The adapters degrade gracefully without a token (REST at 60 req/hr or just readability scraping), but with GITHUB_TOKEN set you get the full rich metadata pipeline — review decisions, diff stats, release assets, threaded discussion replies, and associated PRs on commits — all from one GraphQL query per resource.

    This looks really great! How does it handle stealth such as bot evasion/CAPTCHAs?

[2026-06-22T14:32:25.955000+00:00] magnus919 (id=1518624711712444627)
    GroktoCrawl treats bot evasion as a layered fallback chain rather than a single solution. It tries the least aggressive approach first and escalates only when needed.
    
        For most sites, the first attempt is simple content negotiation -- `/llms.txt` or `Accept: text/markdown` headers. These don't trigger bot detection at all because they're legitimate HTTP standards, and they cover a surprising number of sites (especially developer docs, blogs, and CMS platforms).
    
        When those don't work, it escalates to a headless Chromium instance with Playwright. That browser session runs with realistic fingerprinting: proper navigator.plugins, language settings, hardware concurrency, WebGL vendor strings, and a randomized viewport. It looks like a genuine browser to most anti-bot systems.
    
        If it hits a Cloudflare JS challenge specifically, there's a FlareSolverr integration that solves the challenge and hands back the session cookies.
    
        For the really nasty stuff -- Shopify UCP stores that serve agent-friendly llms.txt instead of the actual page content via content negotiation -- there's a dedicated Shopify adapter that uses a regular browser User-Agent and readability-lxml extraction instead.
    
        The philosophy is: try the cheap path first, escalate deliberately, and never fabricate content when extraction genuinely fails. Everything feeds into a clean markdown output regardless of which tier actually got the data.
