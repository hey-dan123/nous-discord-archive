    Hey everyone -  I’m working on EVA, my small open-source research project around controlled self-modification for LLM agents.
    
    Since this community is building and discussing Hermes Agent, I thought the architecture might be interesting here, motivated by the question:
    
    What would it look like if an agent could modify its own runtime through a gated release process instead of editing itself live?
    
    EVA’s loop is:
    
    active release → candidate → tests/ratchet → kernel gate → promotion → ledger → rollback
    
    The kernel stays small and non-evolving. The evolvable release contains the agent loop, tools, adapters, memory, context compaction, TUI and self-model.
    
    It’s experimental, not production-safe, and runs in Docker with a safe sandbox by default.
    
    I’d love technical feedback on:
    - whether the kernel/release boundary makes sense
    - whether the gates are meaningful or too easy to game
    - whether the seed is already too large
    - how this compares to skill/memory-based growth in Hermes
