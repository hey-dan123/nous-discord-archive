    # Software Architecture Analysis — Codebase Reverse Engineering to Design Document
    
    ## When to Use
    
    - A reference implementation exists and you need to understand its architecture for design inspiration
    - You need a PRD, design document, or specification for a system in the same problem space
    - The output must be clean-room: zero source code samples copied from the reference codebase
    - You're designing a system with different architectural constraints (local-first, privacy-first, self-hosted) than the reference
    - You need to extract an implicit contract — the storage operations a codebase performs — to design a formal provider abstraction

[2026-05-29T23:46:54.801000+00:00] magnus919 (id=1510066942591893625)
    ```
    A kanban-based optimization pipeline that treats skill improvement as a batch process
    
       Map SkillOpt's architecture onto our existing kanban infrastructure: a "Skill Optimization" board with a task per epoch. Each task: (1) roll out the skill against N test tasks (workertask), (2) collect failure patterns (reflection task), (3) propose edits (optimizer task), (4) validate edits against held-out set (validation gate task). The kanban dispatcher managesiteration, and the board provides visibility into the optimization process.
    
       The pipeline architecture already exists — it's the kanban multi-agent system. What's missing is the data flow: a standardized way to capture test task results, compare before/afterperformance, and gate task promotion on improvement. This is closer to SkillOpt's full pipeline than a one-off gate, but also more infrastructure to build.
    ```

    The plan has 6 phases:
    
    Phase  What                                                                                                  Skills Needed
    ─────  ────────────────────────────────────────────────────────────────────────────────────────────────────  ───────────────────────────────────────────
    1      Repo scaffold + SKILL.md — the methodology document, the core artifact                                project-bootstrap, opensource-contributions
    2      seed-board.sh — creates the board, state directory, baseline snapshot, first epoch tasks              kanban-orchestrator
    3      run-phase.sh — unified phase runner covering all 6 pipeline stages with error handling                terminal for scripting patterns
    4      Reference docs — methodology-guide.md, test-suite-design.md, artifact-formats.md, getting-started.md  Writing skills
    5      README + AGENTS.md polish                                                                             project-bootstrap
    6      GitHub push + release + dogfood                                                                       github-pr-workflow or forgejo-cli
    
    Key methodological decisions baked into the plan:
    
    • Training/validation separation is enforced by the seed script — separate arguments, stored in metadata, checked at validate time
    • Edit budget defaults to 4 (paper's Lt=4 optimal), stored in board-metadata.json, user-configurable
    • Rejected edits are never lost — they accumulate in rejected-buffer.json and feed the slow-meta phase
    • Phase order is enforced by the runner — propose without a reflection document errors out with guidance
    • All intermediate artifacts are structured JSON with documented schemas — so the pipeline is inspectable and composeable
