    we've been making remarkable progress on what building continuity across sessions should look like, while maintaining privacy for the user. 
    
    most memory approaches today reach for vector databases and cloud retrieval pipelines. they're powerful, but embedding API cost that scales with usage. recall model trades precision for fuzziness, this becomes especially apparent as memory starts to degrade across long sessions.
    
    our team at Sibyl have been researching a different path. file-based. locally stored. we ran it against the LongMemEval benchmark and scored 95.6% - #2 on the community leaderboard, beating out VC funded memory layers like Hindsight & Mastra. It was the only file-based system in the top tier. every other top 5 system runs on embeddings.
    
    the schema is five tiers: HOT (live state - priorities, treasury, session), WARM (entities and relationships with UNIQUE constraint enforcement), COLD (append-only journal), REFERENCE (runbooks, constants), and ARCHIVE (frozen, recoverable). 
    
    multi-tenant by tenant_id on every row. single source-of-truth enforced at the DB level - a bug cannot create two conflicting facts about the same entity.
    
    full architecture writeup and per-category benchmark breakdown: 
    
    blog.sibylcap.com/longmemeval-v2
    
    our team is currently building the Hermes plugin, and early access will be open next week. if you're hitting memory degradation on long Hermes sessions, this was designed around that exact failure mode. DM me if you'd like to participate as an early tester.
