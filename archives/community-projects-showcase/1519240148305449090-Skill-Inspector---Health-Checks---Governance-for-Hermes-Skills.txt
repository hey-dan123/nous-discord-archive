    Hermes proved that agents can learn.
    
    But while analyzing my own skill library, I realized something:
    
    More skills ≠ better agents.
    
    I ran a governance scan against my Hermes installation and found:
    
    • 70 skill packages
    • 415 assets
    • Only 4 skills had ever been executed
    • 66 skills had never been used
    • Several packages contained 50+ templates, scripts, or references
    
    It reminded me of software engineering:
    
    More code ≠ better software.
    
    Likewise:
    
    More skills ≠ better agents.
    
    As agent ecosystems grow, I think we're going to need more than skill creation.
    
    We'll need skill governance.
    
    To explore this idea, I built Skill Inspector — an open-source governance tool for Hermes skill libraries.
    
    It generates:
    
    • Skill Health Scores
    • Runtime Utilization Reports
    • Duplicate Detection
    • Governance Recommendations
    
    Open Source:
    
    https://github.com/johnsonbuilds/skill-inspector
    
    Learn More:
    
    https://hermesagentcloud.com/skill-inspector
    
    Curious if anyone else has started running into skill sprawl yet.

    The Hermes agent proved that 'learning' is cool, but it has yet to prove that the return on learning outweighs the cost. This, to me, is the core contradiction.
    
    ## Problem #1: Skill Explosion
    
    This is the biggest problem, in my opinion.
    
    The logic behind Hermes sounds great:
    
    Task
    ↓
    Experience
    ↓
    Skill
    ↓
    Store for future reuse
    
    Who wouldn't want that?
    
    But in reality, most experiences aren't valuable enough to become permanent skills.
    
    Imagine this:
    
    Day 1:
    
    The agent fixes an AWS permission issue.
    
    It creates a skill.
    
    Day 2:
    
    The agent fixes an S3 permission issue.
    
    It creates another skill.
    
    Day 3:
    
    The agent fixes an IAM permission issue.
    
    It creates another skill.
    
    A year later?
    
    You might have thousands of skills.
    
    Then a new problem appears:
    
    Which one should be used?
    
    Humans struggle with this problem already.
    
    Why would AI magically solve it?
    
    Anyone who has worked in software engineering has seen this before.
    
    More code does not automatically create more capability.
    
    Very often:
    
    # More code
    
    More maintenance
    
    Skill libraries are no different.
    
    More skills do not necessarily create a smarter agent.
    
    They may simply create a larger system to maintain.

[2026-06-24T07:42:55.761000+00:00] johnsonbuilds (id=1519246432580669490)
    ## Problem #2: Learning Quality Cannot Be Guaranteed
    
    Hermes implicitly assumes:
    
    # Task completed
    
    Correct lesson learned
    
    I don't think that's always true.
    
    Imagine an agent encounters a bug.
    
    It retries three times.
    
    Eventually the task succeeds.
    
    The agent creates a skill:
    
    "If this error occurs, retry three times."
    
    But what if the real reason wasn't the retries?
    
    What if the network simply recovered on its own?
    
    The agent learned a pattern.
    
    But it learned the wrong lesson.
    
    Human organizations already understand this problem.
    
    That's why we have:
    
    * Code Reviews
    * Architects
    * Tech Leads
    
    The purpose of all of them is the same:
    
    Experience needs review.
    
    Not every lesson should become institutional knowledge.
    
    One of the biggest unanswered questions in Hermes is:
    
    Who reviews the skills?

    ## Problem #3: Retrieval Is Harder Than Learning
    
    Many people think learning is the difficult part.
    
    I don't.
    
    I think everything that happens after learning is much harder.
    
    Learning
    ↓
    Storage
    ↓
    Retrieval
    ↓
    Matching
    ↓
    Application
    
    The hardest parts are at the end.
    
    Imagine an agent with 10,000 skills.
    
    A task arrives:
    
    "Deploy a FastAPI application."
    
    Which skill should be selected?
    
    * deploy_python
    * deploy_fastapi
    * deploy_fastapi_aws
    * deploy_backend
    * deploy_api_server
    
    Which one is correct?
    
    This is an extremely difficult search problem.
    
    Many agent projects eventually discover the same thing:
    
    Creating skills is easy.
    
    Finding the right skill is hard.

    ## Problem #4: Diminishing Returns
    
    This is the issue that worries me most.
    
    Imagine the first 100 skills.
    
    Result:
    
    Capability improves by 50%.
    
    Great.
    
    Now imagine the next 1,000 skills.
    
    Result:
    
    Capability improves by 5%.
    
    Still useful.
    
    Now imagine 10,000 skills.
    
    Maybe capability improves by 1%.
    
    Maybe less.
    
    Possibly even negative.
    
    Because as skill count grows:
    
    Search cost
    ↑
    
    Conflict cost
    ↑
    
    Maintenance cost
    ↑
    
    Complexity cost
    ↑
    
    At some point, skill growth can outpace capability growth.
    
    When that happens, more learning stops being an advantage.
    
    It becomes a burden.
